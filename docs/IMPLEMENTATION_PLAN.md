# Grid Engineering 구현 계획

## 1. 기반 원칙

- 대상은 Minecraft 1.21.1, NeoForge, Java 21이다.
- 기계에는 전압 등급을 부여하지 않는다. 전압과 암페어는 전송망에서만 의미가 있다.
- 내부 에너지, 전력, 손실 누계에는 `int`를 사용하지 않는다.
- 일반 틱 연산의 표준 표현은 `long`이다. 덧셈과 곱셈은 `Math.addExact`,
  `Math.multiplyExact` 등으로 오버플로를 검출한다.
- 퍼센트 손실처럼 곱셈 중간값이 `long` 범위를 넘을 수 있는 계산은
  `BigInteger`를 사용하고, 최종 결과만 검증 후 `long`으로 변환한다.
- `long` 범위를 넘는 데이터 정의는 로딩 단계에서 오류로 거부한다. 조용한 포화,
  래핑, `int` 캐스팅은 허용하지 않는다.
- FE는 호환 계층에서만 사용한다. FE capability의 `int` 입출력은 여러 번의 안전한
  청크 전송으로 분할하며, 네트워크 내부 상태는 계속 `long`으로 유지한다.

## 2. 모듈 및 패키지 구조

```text
dev.gridengineering
├─ api                 공개 API와 이벤트
├─ energy              내부 에너지 값, 전압, 암페어, 전력 계산
├─ network             토폴로지, 구간 그래프, 라우팅, 네트워크 관리자
├─ conductor           전선 블록, 블록 엔티티, 데이터 정의
├─ device              변압기, FE 포트, 분석기, 모니터
├─ loss                손실 모델과 레지스트리
├─ data                JSON 로더, 검증기, 동기화
├─ config              서버/클라이언트 설정
├─ persistence         SavedData와 데이터 마이그레이션
└─ client              화면, 오버레이, 렌더링
```

`api`는 구현 패키지에 의존하지 않게 유지한다. 호환 모드는 API 인터페이스와
등록 이벤트만 사용하며 내부 네트워크 클래스에 직접 접근하지 않는다.

## 3. 전력 네트워크 구조

### 월드 관리자

각 `ServerLevel`에 `PowerNetworkManager` 하나를 둔다. 관리자는 다음을 소유한다.

- `networkId -> PowerNetwork`
- `BlockPos.asLong() -> nodeId`
- 청크별 노드 인덱스
- 토폴로지 변경 더티 큐
- 전력 재계산 더티 큐
- 다음 네트워크 ID

클라이언트는 계산 권한을 가지지 않는다. 분석기와 모니터에 필요한 스냅샷만
서버에서 동기화한다.

### 노드와 구간

- `ConductorNode`: 전선, 포트, 변압기 면 등 실제 연결 지점
- `JunctionNode`: 분기점 또는 재질/등급이 바뀌는 지점
- `EndpointNode`: 발전/소비 FE 포트와 Grid Engineering 장치
- `TransmissionSegment`: 두 중요 노드 사이의 직선 또는 비분기 전선 묶음

매 블록을 매 틱 순회하지 않도록, 연속된 같은 전선은 하나의 구간으로 축약한다.
구간에는 길이, 허용 전압, 허용 암페어, 손실 모델, 현재 부하를 캐시한다.

### 병합과 분리

- 전선 설치는 인접 네트워크를 즉시 병합한다.
- 전선 제거는 영향을 받은 기존 네트워크만 분리 후보로 표시한다.
- 분리는 해당 컴포넌트 안에서 예산 제한 BFS/DFS로 처리한다.
- 청크 언로드는 파괴로 취급하지 않는다. 경계 연결을 휴면 상태로 기록한다.
- 토폴로지 변경 중에는 마지막으로 확정된 안전한 스냅샷을 사용하거나 전송을
  잠시 보류한다.

변압기는 양쪽 포트가 서로 다른 유선 네트워크에 속하는 능동 장치다. 따라서
전압 변환이 네트워크 병합을 일으키지 않는다.

## 4. 전선 데이터 구조

전선 종류는 코드에 enum으로 고정하지 않고 데이터 정의로 읽는다.

```text
ConductorDefinition
- id: ResourceLocation
- voltageTier: ResourceLocation
- maxAmperage: long
- lossModel: ResourceLocation
- lossParameters: Map<String, value>
- overcurrentProfile: ResourceLocation 또는 전역 기본값
- appearance: 모델/텍스처 참조
- tags: Set<ResourceLocation>
```

런타임에는 검증을 끝낸 불변 객체만 사용한다. `maxAmperage > 0`, 존재하는 전압
티어, 유효한 손실 모델, 계산 가능한 최대 전력 등을 데이터 로딩 시 검사한다.

동적으로 추가된 전선도 사용할 수 있도록 최종 구조는 “정의 ID를 저장하는 공용
전선 블록/아이템”을 기본으로 한다. 정의 ID는 블록 엔티티와 아이템 데이터
컴포넌트에 저장한다. 내장 전선은 같은 시스템 위에 기본 데이터로 제공한다.

## 5. 전력 계산 방식

### 값 모델

```text
PowerPacket
- voltage: long
- amperage: long
- powerPerTick: long  // multiplyExact(voltage, amperage)
```

에너지는 가능한 한 `powerPerTick`과 틱 수를 분리해 다룬다. 저장 에너지나 누계가
필요한 장치만 `long energy`를 가진다.

### 틱 처리 순서

1. 입력 포트와 변압기가 이번 틱의 공급 가능량을 제시한다.
2. 출력 포트가 수요량과 선택적 우선순위를 제시한다.
3. 캐시된 구간 그래프에서 경로를 선택한다.
4. 경로의 최소 전압 한계와 잔여 암페어를 적용한다.
5. 경로 길이와 구간별 손실을 계산한다.
6. 수신 가능한 양을 FE 장치에 시뮬레이션한다.
7. 실제 추출과 삽입을 커밋한다.
8. 입력, 출력, 손실, 효율, 과전류 상태 스냅샷을 갱신한다.

공급/수요 전체를 전역 최적화하는 비싼 최대 유량 계산은 기본 틱 루프에서 피한다.
초기 버전은 우선순위 + 공정한 라운드 로빈 + 잔여 구간 용량 방식으로 처리한다.
경로는 목적지별로 캐시하고 토폴로지 또는 전선 정의가 바뀔 때만 무효화한다.

### 손실 모델

공개 인터페이스는 다음 형태를 목표로 한다.

```text
LossModel.calculate(inputPower, segmentLength, parameters) -> LossResult
```

기본 구현:

- `percentage`: 기준 블록마다 지정 비율 손실
- `fixed`: 기준 블록마다 지정 고정 전력 손실
- `lossless`: 손실 0

퍼센트는 부동소수점이 아니라 정수 분수(`numerator/denominator`) 또는 basis point로
저장한다. 구간별 반올림 규칙은 설정으로 바꾸지 않고 하나로 고정해 복제성과
테스트 가능성을 보장한다.

### 과전류

경로 구간의 실제 암페어가 허용량을 넘을 때 정책을 적용한다.

- 추가 손실
- 효율 제한
- 내구도 손상
- 전선 파괴
- 무시

파괴/손상은 계산 중 바로 월드를 수정하지 않고 후처리 큐에 넣는다. 동일 틱의
그래프 순회 중 토폴로지가 변하는 문제를 방지하기 위해서다.

## 6. 저장 방식

- 전선 정의 ID, 손상도, 장치 설정은 블록 엔티티에 저장한다.
- 무선/차원 간 링크, 안정적인 ID 발급기, 필요한 전역 상태는 `SavedData`에 저장한다.
- 일반 유선 네트워크의 전체 그래프는 저장하지 않는다. 로드된 청크의 블록과
  청크 인덱스로 재구성해 오래된 위치 데이터가 남는 문제를 피한다.
- 네트워크 ID는 진단용이며 영구적인 게임 규칙 식별자로 사용하지 않는다.
- 저장 데이터에는 스키마 버전을 넣고 DataFixer와 별개의 명시적 마이그레이션
  단계를 둔다.

## 7. 성능 최적화 전략

- 블록 설치/제거, 이웃 변경, 청크 로드/언로드 이벤트로만 토폴로지를 더럽힌다.
- 네트워크 전체를 매 틱 탐색하지 않는다.
- 연속 전선을 구간으로 축약하고, 경로와 네트워크 통계를 캐시한다.
- 토폴로지 재구성에는 틱당 노드/시간 예산을 둔다.
- 전력이 흐르지 않고 상태가 변하지 않은 네트워크는 휴면 처리한다.
- FE capability 검색 결과는 이웃 변경 시에만 무효화한다.
- 원시 키 컬렉션과 `BlockPos.asLong()`을 사용해 객체 할당을 줄인다.
- 모니터 패킷은 변경량 제한, 주기 제한, 관찰 중인 플레이어 기준으로 전송한다.
- 프로파일링 지표로 처리 네트워크 수, 재구성 노드 수, 경로 캐시 적중률,
  전송량과 패킷량을 노출한다.

## 8. API 구조

초기 공개 API 후보:

- `GridEngineeringAPI`
- `VoltageTier`
- `ConductorDefinition`
- `GridPowerPort`
- `PowerOffer` / `PowerDemand`
- `LossModel`
- `LossModelType`
- `RegisterGridEngineeringTypesEvent`
- 네트워크 상태 조회용 읽기 전용 `PowerNetworkView`

API에는 NeoForge capability를 제공해 사용자 정의 장치가 공급/수요 포트로 참여할
수 있게 한다. FE 어댑터는 별도 구현이며 내부 포트 API가 FE 타입에 의존하지 않게
한다.

호환성 원칙:

- 공개 타입은 가능한 한 불변 값 객체다.
- 등록 ID는 `ResourceLocation`을 사용한다.
- 구현 객체를 반환하지 않고 읽기 전용 뷰를 반환한다.
- API 버전을 모드 버전과 별도로 명시한다.
- 계산 이벤트는 취소/수정 범위를 제한해 서버 성능과 결정성을 보호한다.

## 9. 설정 파일 구조

설정은 역할에 따라 나눈다.

### NeoForge 서버 설정

월드별 `serverconfig/gridengineering-server.toml`:

- 기본 과전류 정책
- 토폴로지 재계산 예산
- 네트워크당 전송 작업 예산
- FE 입출력 청크/호출 제한
- 모니터 갱신 주기
- 전선 손상 및 파괴 허용 여부
- 양자/차원 간 전송의 운영 제한

클라이언트 설정:

- 분석기 표시 형식
- 오버레이 위치와 갱신 빈도
- 전선 상태 렌더링 품질

### 데이터 기반 설정

데이터팩 JSON:

```text
data/<namespace>/gridengineering/voltage_tier/*.json
data/<namespace>/gridengineering/conductor/*.json
data/<namespace>/gridengineering/overcurrent_profile/*.json
data/<namespace>/gridengineering/device_profile/*.json
```

전압 티어와 전선 추가는 JSON이 담당한다. 서버 TOML은 운영 정책만 담당한다.
이 구분은 목록형 콘텐츠를 TOML에 억지로 넣지 않고 데이터팩 재정의, 리소스팩,
서버 동기화를 활용하기 위한 것이다.

기본 전압:

| ID | 표시명 | 전압 |
|---|---|---:|
| `lv` | Low Voltage | 65,536 |
| `mv` | Medium Voltage | 524,288 |
| `hv` | High Voltage | 4,194,304 |
| `shv` | Super High Voltage | 268,435,456 |
| `chv` | Critical High Voltage | 2,147,483,648 |

## 10. 데이터 기반 확장과 동기화

- 서버 시작/데이터팩 리로드 시 JSON을 파싱하고 전체 참조를 검증한다.
- 오류가 있는 정의는 파일 경로와 필드를 포함한 명확한 오류를 남긴다.
- 서버가 승인한 정의의 압축 스냅샷을 접속 클라이언트에 전송한다.
- 클라이언트는 서버 정의를 UI와 렌더링에만 사용한다.
- 리로드로 정의가 사라진 전선은 즉시 월드를 망가뜨리지 않고
  `missing definition` 안전 상태가 되어 전송을 중단한다.
- 손실 모델과 장치 종류는 코드/API 등록, 실제 수치와 인스턴스는 JSON 등록으로
  분리한다.

## 11. 구현 단계

### 단계 0 — 프로젝트 기반

- NeoForge MDK, Java 21, IntelliJ/Gradle 구성
- 모드 진입점과 메타데이터
- 설계 문서와 CI 빌드

### 단계 1 — 수치 및 데이터 계층

- `long` 기반 값 객체와 오버플로 테스트
- 전압 티어/전선/손실 JSON 로더
- 기본 데이터와 검증 오류 보고

### 단계 2 — 전선과 토폴로지

- 공용 전선 블록/아이템/블록 엔티티
- 연결 규칙, 청크 인덱스, 네트워크 병합/분리
- 구간 축약과 네트워크 통계

### 단계 3 — FE 경계와 전송

- 입력/출력 FE 포트
- 공급/수요 수집, 암페어 용량, 기본 라우팅
- 퍼센트/고정/무손실 모델

### 단계 4 — 플레이어 피드백

- 전력 분석기
- 네트워크 상태 스냅샷과 모니터 UI
- 디버그 명령과 프로파일링 지표

### 단계 5 — 위험과 변환

- 과전류 정책과 전선 손상
- 변압기
- 상세 전송 테스트와 서버 부하 테스트

- 공개 API 문서, 호환성 테스트, 마이그레이션 정책

## 12. 검증 계획

- 단위 테스트: 오버플로, 전력 공식, 손실 반올림, 암페어 제한
- 속성 기반 테스트: 입력보다 출력+손실이 커지지 않는지 검증
- GameTest: 설치/제거/분리/병합, 청크 경계, 변압기
- 호환 테스트: 대표적인 FE 발전기, 저장고, 소비 장치
- 전용 서버 테스트: 클라이언트 클래스 로딩 누출 방지
- 부하 테스트: 수천~수만 전선, 다수 독립망, 반복 청크 로드
- 저장/재시작/데이터팩 리로드 및 누락 정의 복구 테스트
